# 02 — Development Guide (Recipes)

Step-by-step recipes. Every example mirrors the existing **Auth** feature — use it as a living
template. Read [01_ARCHITECTURE.md](01_ARCHITECTURE.md) and [CLAUDE.md](../CLAUDE.md) first.

Build from the database outward: **migration → entity → repository → service → DTOs → controller → test**.

---

## Recipe 0 — Generate a feature, then fill it in (the fast path)

Don't hand-write the boilerplate — stamp it and edit the fields:

```bash
scripts/new-feature.sh Criterion criteria
```

You get a compiling, tested, paged-CRUD package (`criterion/`), a Flyway migration with the next
version number, and endpoints under `/v1/criteria`. It uses a single placeholder `name` column.
Then:

1. Open `criterion/Criterion.java` and replace/extend `name` with your real columns (e.g. `weight`,
   `type`). Add getters/setters.
2. Mirror those columns in the generated `V<n>__create_criteria.sql` migration.
3. Update `dto/CriterionRequest.java` (+ validation) and `dto/CriterionResponse.java` (+ `from`).
4. `./mvnw test`.

Reusable pieces the generated code (and you) should use: `BaseEntity` (id + timestamps),
`PageResponse<T>` (list envelope), and the `common/exception/` types. Recipe A below shows the
same result written by hand, for when you need something the generator doesn't cover.

## Recipe A — Add a new feature (end-to-end, by hand)

Example: a **`criteria`** feature (SPK criteria CRUD). Replace `criteria`/`Criterion` with your
feature name. Create the package `com.spkmainan.criteria` with a `dto/` sub-package.

### 1. Migration — `src/main/resources/db/migration/V2__init_criteria.sql`
```sql
CREATE TABLE criteria (
    id         BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name       VARCHAR(255) NOT NULL,
    weight     NUMERIC(5,4) NOT NULL,          -- 0.0000 .. 1.0000
    type       VARCHAR(20)  NOT NULL,          -- BENEFIT | COST
    created_at TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```
Never edit an applied migration — always add the next `V<n>__*.sql`.

### 2. Entity — `criteria/Criterion.java`
Extend `BaseEntity` — it already provides `id`, `createdAt`, `updatedAt`, so only declare the
feature's own columns.
```java
@Entity
@Table(name = "criteria")
public class Criterion extends BaseEntity {

    @Column(nullable = false)
    private String name;

    @Column(nullable = false, precision = 5, scale = 4)
    private BigDecimal weight;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private CriterionType type;

    protected Criterion() { } // JPA

    public Criterion(String name, BigDecimal weight, CriterionType type) {
        this.name = name;
        this.weight = weight;
        this.type = type;
    }

    // getters/setters (no Lombok)
}
```

### 3. Repository — `criteria/CriterionRepository.java`
```java
public interface CriterionRepository extends JpaRepository<Criterion, Long> {
    boolean existsByNameIgnoreCase(String name);
}
```

### 4. DTOs — `criteria/dto/`
```java
// CriterionRequest.java — input, validated
public record CriterionRequest(
        @NotBlank String name,
        @NotNull @DecimalMin("0.0") @DecimalMax("1.0") BigDecimal weight,
        @NotNull CriterionType type) { }

// CriterionResponse.java — output
public record CriterionResponse(String id, String name, BigDecimal weight, String type) {
    public static CriterionResponse from(Criterion c) {
        return new CriterionResponse(
                String.valueOf(c.getId()), c.getName(), c.getWeight(), c.getType().name());
    }
}
```

### 5. Service — `criteria/CriterionService.java`
```java
@Service
public class CriterionService {

    private final CriterionRepository repository;

    public CriterionService(CriterionRepository repository) {
        this.repository = repository;
    }

    @Transactional(readOnly = true)
    public List<CriterionResponse> findAll() {
        return repository.findAll().stream().map(CriterionResponse::from).toList();
    }

    @Transactional(readOnly = true)
    public CriterionResponse findById(Long id) {
        return CriterionResponse.from(getOrThrow(id));
    }

    @Transactional
    public CriterionResponse create(CriterionRequest request) {
        Criterion c = new Criterion(request.name(), request.weight(), request.type());
        return CriterionResponse.from(repository.save(c));
    }

    private Criterion getOrThrow(Long id) {
        return repository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException("Criterion not found: " + id));
    }
}
```

### 6. Controller — `criteria/CriterionController.java`
```java
@RestController
@RequestMapping("/criteria")
public class CriterionController {

    private final CriterionService service;

    public CriterionController(CriterionService service) {
        this.service = service;
    }

    @GetMapping
    public List<CriterionResponse> list() {
        return service.findAll();
    }

    @GetMapping("/{id}")
    public CriterionResponse get(@PathVariable Long id) {
        return service.findById(id);
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public CriterionResponse create(@Valid @RequestBody CriterionRequest request) {
        return service.create(request);
    }
}
```
The endpoint is authenticated by default (no `SecurityConfig` change needed). To restrict to
admins, annotate with `@PreAuthorize("hasRole('ADMIN')")` and ensure method security is enabled.

### 7. Test — `src/test/java/com/spkmainan/criteria/CriterionControllerTest.java`
Mirror `AuthControllerTest`: one happy path, one failure (e.g. validation 400, not-found 404).
Authenticate protected calls with `@WithMockUser` or by logging in to obtain a token.

### 8. Done check
`./mvnw test` green. Update [05_ROADMAP.md](05_ROADMAP.md) and, if the endpoint is client-facing,
[04_API_REFERENCE.md](04_API_REFERENCE.md).

## Recipe B — Add an endpoint to an existing feature

1. Add a method to the service (with the right `@Transactional`).
2. Add request/response DTO records if the shape is new.
3. Add the controller method; validate input with `@Valid`.
4. Add a test for it.

No `SecurityConfig` change unless the endpoint must be public — then add its path to `PUBLIC_PATHS`.

---

## Recipe C — Add a database migration

1. Create the next file: `db/migration/V<n>__<snake_desc>.sql` (n strictly increments).
2. Write portable SQL (H2 v2 + PostgreSQL). Prefer ANSI types.
3. Update the affected entity so its columns match (JPA `validate` runs on startup).
4. `./mvnw test` — Flyway applies migrations against H2 during tests and fails fast on mismatch.

Never modify a migration that has already been applied anywhere — add a new one.

---

## Recipe D — Make an endpoint public / role-restricted

- **Public:** add the path (relative to `/v1`) to `PUBLIC_PATHS` in `config/SecurityConfig`.
- **Role-restricted:** use `@PreAuthorize("hasRole('ADMIN')")` on the method (authorities are
  `ROLE_<NAME>`), or an `.requestMatchers(...).hasRole(...)` rule in the filter chain.

---

## Common pitfalls

- **JPA validation fails on startup** → entity column name/type ≠ migration. Align them.
- **403/401 on a new endpoint** → it's authenticated by default; send a bearer token or make it public.
- **`avatar_url` etc. serialized as camelCase** → add `@JsonProperty("avatar_url")` on the DTO field.
- **Editing an applied migration** → Flyway checksum error. Add a new migration instead.
